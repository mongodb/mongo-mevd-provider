# MongoDB.UnitTests — AGENTS.md

The unit-test project. Lives at `test/MongoDB.UnitTests/`. No MongoDB connection required — every test mocks `IMongoDatabase` / `IMongoCollection<BsonDocument>` via Moq. Suitable for fast turnaround on mapping logic, filter-translation BSON shape, DI registration plumbing, and any pure logic in `MongoCollection` / `MongoVectorStore` that can be exercised against mocked Mongo abstractions.

## Scope

In: mapper round-trips (`MongoMapperTests`, `MongoDynamicMapperTests`), `MongoVectorStore` / `MongoCollection` operations whose only Mongo touchpoint is the mocked collection/database, model-builder validation, filter-translator output (BSON-document equality).

Out: anything that needs an actual Atlas-search index, real `$vectorSearch` / `$search` results, or Testcontainers — those live in `test/MongoDB.ConformanceTests/`. Don't push Atlas-dependent assertions into here.

## Conventions

- **Mocking pattern.** `_mockMongoDatabase.Setup(d => d.GetCollection<BsonDocument>(It.IsAny<string>(), It.IsAny<MongoCollectionSettings>())).Returns(_mockMongoCollection.Object)` in the test-class ctor wires the database to return a single shared mock collection. Then per-test `Setup` calls on `_mockMongoCollection` configure `ReplaceOneAsync` / `FindAsync` / `AggregateAsync` / `DeleteOneAsync` etc. as needed. Verify with `_mockMongoCollection.Verify(...)`.
- **Test models** live alongside the tests (`MongoHotelModel.cs`). They use MEVD attributes (`[VectorStoreKey]`, `[VectorStoreData]`, `[VectorStoreVector]`) and occasionally BSON attributes (`[BsonId]`, `[BsonElement("...")]`) to cover the storage-name interplay. Keep model classes small — one model per scenario is fine; don't try to make one super-model cover every case.
- **Sync over async in arrangement only.** Test bodies are `async Task` whenever a method-under-test is async (which is most of `MongoCollection`). Use `await` rather than `.Result` / `.GetAwaiter().GetResult()`.
- **Single MEVD-namespace alias.** When the test needs both `Microsoft.Extensions.VectorData.VectorSearchOptions<T>` and a local `VectorSearchOptions`, the existing pattern is `using MEVD = Microsoft.Extensions.VectorData;` and full-qualifying with `MEVD.VectorSearchOptions<T>`.
- **`InternalsVisibleTo`.** `MongoDB.VectorData.UnitTests` is in the `InternalsVisibleTo` list on the main project (see `src/MongoDB/MongoDB.csproj`). Internals are reachable; use them deliberately — preferring public surface where it covers the case, so unit tests can serve as documentation for the public API.
- **Frameworks.** Unit tests target `net10.0` only (see `test/Directory.Build.props` / `MongoDB.UnitTests.csproj`). The provider multi-targets `net10`/`net8`/`netstandard2.1`/`net472` but the unit-test suite exercises the runtime-current behavior.

## Common pitfalls

- **Mocking `IAsyncCursor<T>`.** Most search / get methods iterate an `IAsyncCursor<T>`. The minimal mock pattern is: `cursor.SetupSequence(c => c.MoveNextAsync(default)).ReturnsAsync(true).ReturnsAsync(false); cursor.Setup(c => c.Current).Returns(batch);`. Forgetting the trailing `false` produces an infinite loop in the test. Don't add real `Task.Delay`s to "wait for state" — use deterministic mock sequences.
- **Asserting BSON equality.** `BsonDocument.Equals` is value-equality on element name + value, so prefer `Assert.Equal(expected, actual)` over `Assert.True(expected.Equals(actual))` (the former produces a useful diff message). Field-order matters in pipeline-stage documents; if a test seems wrong, dump `actual.ToJson()` and compare.
- **Auto-generated keys.** `MongoCollection.UpsertCoreAsync` mutates the input record's key when `IsAutoGenerated && value == empty`. A test that re-uses the same record across upsert calls will see the mutation persist — copy or reset.
- **Don't test private state through reflection.** If a test needs to verify internal state, expose it via an `internal` member and use the `InternalsVisibleTo`. The `MongoCollectionTestHook` `Func`-resolver pattern is the established mechanism for test-only escape hatches on otherwise-encapsulated state — extend it rather than reflecting.

## How to run

```bash
dotnet test test/MongoDB.UnitTests
# Single class
dotnet test test/MongoDB.UnitTests --filter "FullyQualifiedName~MongoMapperTests"
# Single method
dotnet test test/MongoDB.UnitTests --filter "FullyQualifiedName~MongoMapperTests.MapFromDataToStorageModel"
```

A failing unit test is the cheapest signal in the repo — keep them green before pushing.
